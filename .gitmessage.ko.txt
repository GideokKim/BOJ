# 커밋 메시지 형식
# ================
#
# Git 커밋 메시지는 매우 정확한 규칙에 따라 형식화되어야 합니다.
# 이 형식은 커밋 히스토리를 더 읽기 쉽게 만듭니다.
#
# 각 커밋 메시지는 헤더, 본문, 푸터로 구성됩니다.
#
# <header>
# <BLANK LINE>
# <body>
# <BLANK LINE>
# <footer>
#
# 헤더는 필수입니다.
#
# 본문은 "docs" 타입을 제외한 모든 커밋에 필수입니다. 본문이 있을
# 경우 최소 20자 이상이어야 합니다.
#
# 푸터는 선택 사항입니다.
#
# 커밋 메시지의 모든 줄은 80자를 초과할 수 없습니다.
#
# 커밋 메시지 헤더
# ----------------
#
# ```
# <type>(<scope>): <short summary>
#   │       │             │
#   │       │             └─⫸ 현재형으로 요약. 대문자로 시작하지 않음.
#   │       │                 끝에 마침표 없음.
#   │       │
#   │       └─⫸ 커밋 범위: algebra|combinatorics|data_structure|geometry|
#   │                     number_theory|problem|test|util
#   │
#   └─⫸ 커밋 타입: build|chore|ci|docs|feat|fix|perf|refac|style|test
# ```
#
# 커밋 메시지 본문
# ----------------
#
# 요약과 마찬가지로 명령형 현재형을 사용합니다: "fix"를 사용하고
# "fixed"나 "fixes"는 사용하지 않습니다.
#
# 커밋 메시지 본문에서 변경의 동기를 설명하세요. 이 커밋 메시지는
# 왜 변경을 하는지 설명해야 합니다. 변경의 영향을 설명하기 위해
# 이전 동작과 새 동작을 비교할 수 있습니다.
#
# 커밋 메시지 푸터
# ----------------
#
# 푸터는 사용법에 대한 정보를 포함할 수 있으며, 이 커밋이 닫거나
# 관련된 GitHub 이슈, yona 링크, 기타 PR을 참조하는 곳이기도 합니다.
#
# ```
# USAGE: <usage summary>
# <BLANK LINE>
# <usage description + instruction example>
# <BLANK LINE>
# See #<reference>
# <BLANK LINE>
# Resolves|Related #<issue number>
# ```
#
# USAGE 섹션은 "USAGE: "라는 구문으로 시작하고, 그 다음 사용법
# 요약, 빈 줄, 그리고 사용법에 대한 자세한 설명이 와야 합니다.
